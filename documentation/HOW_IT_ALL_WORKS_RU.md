# Как всё работает: от твоего кода до публикации пакета

## Часть 1. ГДЕ ЖИВЁТ КОД

У твоего проекта два "места жительства":

```
┌──────────────────────┐         git push          ┌──────────────────────┐
│   ТВОЙ КОМПЬЮТЕР     │  ───────────────────────> │      GITHUB          │
│   /home/emil/cicd/   │                           │  (облачный сервер)   │
│                      │  <─────────────────────── │                      │
│   Тут ты пишешь код  │         git pull          │  Тут код хранится    │
│   в VS Code          │                           │  и проверяется       │
└──────────────────────┘                           └──────────────────────┘
```

**Локально (твой компьютер):** Здесь ты открываешь VS Code, редактируешь файлы, сохраняешь.
Пока ты просто сохраняешь файл — GitHub об этом НЕ ЗНАЕТ. Ноль реакции.

**Удалённо (GitHub):** Облачный сервер, где хранится "официальная" копия проекта.
Именно там живут GitHub Actions — роботы, которые проверяют твой код.

---

## Часть 2. ЧТО ТАКОЕ ВЕТКИ

Представь, что `main` — это чистовая тетрадь, а ветка — это черновик.

```
main:           ●───●───●───●  (тут всё стабильно и работает)
                         \
feature/test:             ●───●───●  (тут ты экспериментируешь)
```

- **`main`** = стабильный, рабочий код. Сюда НЕЛЬЗЯ пушить напрямую (мы это заблокировали через Branch Protection).
- **Твоя ветка** (например `feature/test-ci`) = черновик, где ты делаешь что хочешь.

Создаёшь ветку так:
```bash
git checkout -b feature/test-ci
```

Эта команда говорит: "Скопируй текущее состояние `main` в новую ветку `feature/test-ci` и переключись на неё".

---

## Часть 3. ЧТО ТАКОЕ PULL REQUEST

**Pull Request (PR)** — это ЗАЯВКА: "Я хочу перенести изменения из моей ветки-черновика в `main`".

Это **не команда в терминале**. Это **страница на сайте GitHub**, которую ты создаёшь руками.

### Как твой код связан с PR?

Вот цепочка:

```
1. Ты меняешь файл           →  Файл изменён ТОЛЬКО на диске
   (сохранил в VS Code)

2. git add .                  →  Ты говоришь Git: "Запомни эти изменения"

3. git commit -m "message"    →  Git создаёт "снимок" изменений (коммит)
                                  Но всё это пока ТОЛЬКО у тебя на компе!

4. git push origin feature/   →  Коммит улетает на GitHub.
   test-ci                       Теперь GitHub знает о твоей ветке.

5. На сайте GitHub:           →  Ты создаёшь Pull Request.
   "Create Pull Request"         Это и есть заявка: "влейте мою ветку в main".
```

**Ключевой момент:** Без `git push` GitHub не знает о твоём коде. Без Pull Request — роботы (Actions) не запустятся.

## Часть 4. ПОЧЕМУ Я ВИЖУ ПАЙПЛАЙН В ACTIONS ТОЛЬКО ПОТОМ

Роботы GitHub Actions — это программы, которые сидят на серверах GitHub и ЖДУТ события (триггера).

Мы настроили триггеры вот так (файлы `.github/workflows/*.yml`):

```yaml
# pr-checks.yml
on:
  pull_request:        # ← Триггер: "Запустись, когда появится PR"
    branches: [main]
```

**Последовательность такая:**

```
Ты пишешь код             Тишина. Роботы спят.
     ↓
git add + commit          Тишина. Роботы спят.
     ↓
git push                  Тишина. Роботы спят.
                          (push в feature-ветку — не триггер)
     ↓
Создаёшь Pull Request     ⚡ БАМ! Робот проснулся!
на сайте GitHub            GitHub видит событие "pull_request"
                           и запускает pr-checks.yml
     ↓
Вкладка Actions            Тут ты видишь свой пайплайн
на GitHub загорается       с шагами: Lint → Build → Test
```

**Почему не раньше?** Потому что мы написали `on: pull_request`. Это значит "запускайся ТОЛЬКО когда создан или обновлён PR". Обычный `push` в ветку — это не Pull Request, поэтому робот молчит.

## Часть 5. ПОЛНЫЙ ПУТЬ КОДА — ОТ НАЧАЛА ДО КОНЦА

Теперь проходим весь маршрут целиком. Каждый шаг — это то, что реально происходит.

### Этап 1: Ты пишешь код

```bash
# Переключаешься на новую ветку
git checkout -b feature/add-divide

# Открываешь src/index.ts и добавляешь функцию divide()
# Открываешь package.json и меняешь version: "1.0.0" → "1.1.0"
```

Сейчас изменения есть ТОЛЬКО на твоём компьютере.
GitHub ничего не знает. Actions не запускаются.

### Этап 2: Отправляешь код на GitHub

```bash
git add .
git commit -m "feat: add divide function"
git push origin feature/add-divide
```

Теперь GitHub знает о ветке `feature/add-divide`.
Но Actions ВСЁ ЕЩЁ не запускаются — нет триггера (нет PR).

### Этап 3: Создаёшь Pull Request

Заходишь на https://github.com/emilmamedov-inno/ts-math-utils
GitHub сам предложит: "You recently pushed `feature/add-divide`. Create a Pull Request?"
Нажимаешь кнопку. Пишешь описание.

**⚡ Триггер сработал!**

```
GitHub видит: событие "pull_request" → ищет файлы в .github/workflows/
Находит: pr-checks.yml → on: pull_request → ЗАПУСК!
```

**Что делает робот pr-checks.yml:**
```
1. Берёт виртуальную машину Ubuntu на сервере GitHub
2. Скачивает туда твой код из PR
3. Устанавливает Node.js (через наш reusable action setup-node-deps)
4. npm ci — ставит зависимости из package-lock.json
5. Проверяет: есть ли package-lock.json? (если нет — ❌)
6. npm run lint — проверяет код линтером ESLint
7. npm run build — компилирует TypeScript
8. npm test — запускает юнит-тесты Vitest
```

Если хоть один шаг падает — весь пайплайн красный.
Кнопка "Merge" на PR заблокирована (Branch Protection не даёт).

### Этап 4: Лейбл `verify`

Ты заходишь на страницу своего PR → справа панель "Labels" → выбираешь `verify`.

**⚡ Новый триггер!**

```
GitHub видит: на PR повесили лейбл → проверяет workflows
label-verify.yml → if: github.event.label.name == 'verify' → ЗАПУСК!
```

**Что делает робот label-verify.yml:**
```
1. Скачивает твой код
2. Устанавливает зависимости
3. npm run build — собирает пакет
4. npm run test:e2e — запускает E2E тесты
   (они импортируют собранный пакет из dist/ и проверяют,
    что он реально работает как npm-модуль)
```

### Этап 5: Лейбл `publish`

Ты вешаешь лейбл `publish` на тот же PR.

**⚡ Ещё один триггер!**

```
label-publish.yml → if: github.event.label.name == 'publish' → ЗАПУСК!
```

**Что делает робот label-publish.yml:**
```
1. Читает version из package.json → "1.1.0"
2. Проверяет: существует ли 1.1.0 в npm?
   - Если ДА → ❌ ОШИБКА. Ты забыл поднять версию.
   - Если НЕТ → ✅ Продолжаем
3. Генерирует dev-версию: "1.1.0-dev-a3b8c2f" (добавляет хэш коммита)
4. npm run build — собирает пакет
5. npm pack — упаковывает в архив .tgz
6. Загружает .tgz как артефакт (можно скачать со страницы Actions)
```

Это "Release Candidate" — кандидат на публикацию. Можно скачать и проверить.

### Этап 6: Merge (Слияние)

Все проверки зелёные ✅. Коллега нажал "Approve". Ты нажимаешь **"Squash and Merge"**.

Что происходит:
```
feature/add-divide     main
        ●───●───●  ──→  ●  (твои изменения влились в main)
                         ↑
              Pull Request закрылся со статусом "merged"
```

**⚡ Финальный триггер!**

```
release.yml → on: pull_request: types: [closed]
           → if: merged == true AND label == 'publish'
           → ЗАПУСК!
```

### Этап 7: Автоматический релиз

**Что делает робот release.yml:**
```
1. Скачивает код из main (уже с твоими изменениями)
2. npm run build — собирает пакет
3. npm publish — публикует пакет в npm
   (теперь любой человек может сделать:
    npm install @emilmamedov-inno/ts-math-utils)
4. Создаёт Git Tag: v1.1.0
5. Создаёт GitHub Release — красивая страница с описанием
```

**ВСЁ. Код прошёл весь путь:**
```
Твой VS Code → git push → Pull Request → Проверки (lint/build/test) →
→ verify (E2E) → publish (RC) → Merge → npm publish + Git Tag + Release
```

## Часть 6. А ГДЕ ВО ВСЁМ ЭТОМ REUSABLE ACTIONS?

У нас есть ДВА репозитория:

```
emilmamedov-inno/ts-math-utils          ← Основной проект
emilmamedov-inno/cicd-shared-actions    ← Библиотека переиспользуемых действий
```

В `cicd-shared-actions` лежат 3 маленьких "кирпичика":
- **setup-node-deps** — установить Node.js + зависимости
- **check-version-exists** — проверить, есть ли версия в npm
- **create-release** — создать тег + релиз

Зачем? Чтобы не копипастить одни и те же шаги во все 4 workflow файла.
Вместо того чтобы писать 10 строк настройки Node.js в каждом файле, мы пишем одну:

```yaml
- uses: emilmamedov-inno/cicd-shared-actions/setup-node-deps@main
```

GitHub сам идёт в тот репозиторий, скачивает `action.yml` и выполняет шаги, описанные внутри.

## Часть 8. КОРОТКО, ОДНИМ АБЗАЦЕМ

Ты пишешь код на своём компьютере и отправляешь его на GitHub через `git push`. Потом на сайте GitHub ты создаёшь Pull Request — заявку на перенос кода в главную ветку `main`. В момент создания PR, GitHub читает файлы из `.github/workflows/` и запускает роботов (Actions), которые проверяют твой код (линтер, сборка, тесты). Именно поэтому ты видишь пайплайн "только потом" — роботы срабатывают на СОБЫТИЕ (создание PR), а не на факт сохранения файла у тебя на компе. Дальше ты вешаешь лейблы (`verify`, `publish`) — каждый запускает свой workflow. Когда ты нажимаешь Merge — код вливается в `main`, и финальный робот автоматически публикует пакет в npm, создаёт тег и релиз. Никаких ручных шагов после мерджа.
