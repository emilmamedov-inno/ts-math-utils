# ОТВЕТЫ

---

### 1. Что такое GitHub Actions и для чего он используется?
GitHub Actions — встроенная платформа автоматизации GitHub. Позволяет запускать код (скрипты, тесты, деплой) в ответ на события в репозитории (push, PR, лейбл, merge). Используется для CI/CD: автоматической проверки, сборки и публикации кода.

### 2. Что такое Workflow, Job и Step?
- **Workflow** — YAML-файл в `.github/workflows/`, описывающий набор автоматизаций. Один файл = один workflow.
- **Job** — единица работы внутри workflow. Запускается на отдельном Runner-е (VM). Несколько job-ов могут работать параллельно.
- **Step** — одна команда или action внутри job. Шаги выполняются последовательно.

### 3. Где хранятся файлы workflow?
В директории `.github/workflows/` в корне репозитория. GitHub автоматически обнаруживает все `.yml` / `.yaml` файлы в этой папке.

### 4. Что такое Runner?
Runner — виртуальная машина, на которой выполняется workflow. GitHub предоставляет hosted runners (в облаке Azure). VM создаётся с нуля для каждого запуска и УНИЧТОЖАЕТСЯ после завершения — для безопасности и чистоты.

### 5. Что означает `runs-on: ubuntu-latest`?
Указывает, что job должен запускаться на GitHub-hosted Runner с Ubuntu (последняя стабильная LTS-версия). Доступны также `windows-latest` и `macos-latest`.

### 6. Типы событий (triggers)?
`push` — коммит в ветку. `pull_request` — создание/обновление PR. `pull_request: types: [labeled]` — навешивание лейбла. `pull_request: types: [closed]` — закрытие PR. Также: `workflow_dispatch` (ручной запуск), `schedule` (по расписанию, cron), `release`, `issues` и т.д.

### 7. Что будет без `types:` у `on: pull_request`?
По умолчанию workflow сработает на: `opened` (PR создан), `synchronize` (новый коммит в ветку PR), `reopened` (PR переоткрыт). НЕ сработает на `labeled`, `closed` и другие.

### 8. Разница `types: [opened, synchronize]` vs `types: [labeled]`?
Первый запускается при создании PR и при каждом push в его ветку. Второй — ТОЛЬКО когда на PR вешают лейбл. Это разные события, позволяющие запускать разные проверки в разные моменты жизни PR.

### 9. Почему `types: [closed]`, а не `types: [merged]`?
В GitHub API НЕТ типа события `merged`. Merge — это частный случай закрытия. Поэтому мы ловим `closed` и дополнительно проверяем: `if: github.event.pull_request.merged == true`.

### 10. Что такое Webhook event?
Когда происходит событие (PR, лейбл и т.д.), GitHub отправляет HTTP POST запрос (webhook) на Runner с JSON-объектом, содержащим всю информацию о событии: тип, автор, SHA коммита, лейблы, ветки.

### 11. Запустится ли `label-verify.yml` при лейбле `bug`?
Workflow ЗАПУСТИТСЯ (триггер `types: [labeled]` не фильтрует по имени лейбла). Но job будет ПРОПУЩЕН (skipped), потому что условие `if: github.event.label.name == 'verify'` не сработает.

### 12. Что делает `uses:`?
Подключает готовый action (многоразовый блок автоматизации). Может быть из маркетплейса (`actions/checkout@v4`), из другого репозитория (`owner/repo/path@ref`), или локальный (`./my-action`).

### 13. Синтаксис `uses: emilmamedov-inno/cicd-shared-actions/setup-node-deps@main`?
`emilmamedov-inno` — владелец (GitHub аккаунт). `cicd-shared-actions` — репозиторий. `setup-node-deps` — путь к папке с `action.yml` внутри репо. `@main` — ветка (можно тег `@v1` или SHA коммита).

### 14. Что такое Composite Action?
Action, состоящий из набора обычных шагов (shell-команды и `uses:`). Определяется через `using: 'composite'`. В отличие от JavaScript Action (Node.js скрипт) и Docker Action (контейнер), не требует ни кода, ни Docker.

### 15. Зачем отдельный репозиторий?
1) Переиспользуемость: любой проект может подключить через `uses:`. 2) Единая точка обновления: поправил action в одном месте — все проекты получили обновление. 3) Задание требует: «put them in a reusable repo».

### 16. Что такое `$GITHUB_OUTPUT`?
Механизм передачи данных между шагами workflow. Каждый шаг — отдельный процесс. Чтобы шаг 1 передал значение шагу 2, он пишет `echo "key=value" >> $GITHUB_OUTPUT`. Следующий шаг читает через `${{ steps.<id>.outputs.key }}`.

### 17. Чем лучше `::set-output`?
`::set-output` был уязвим к инъекции через stdout (спецсимволы в значении могли выполниться как команды). `$GITHUB_OUTPUT` пишет в файл, который парсится безопасно.

### 18. Что такое `upload-artifact`?
Action, загружающий файлы на облачное хранилище GitHub (Azure Blob Storage). Артефакт доступен для скачивания через UI на странице workflow run. Хранится 7 дней (по умолчанию, до 90).

### 19. `GITHUB_TOKEN` vs `NPM_TOKEN`?
`GITHUB_TOKEN` — создаётся АВТОМАТИЧЕСКИ для каждого workflow, живёт пока workflow работает, даёт доступ только к текущему репозиторию. `NPM_TOKEN` — создаётся ВРУЧНУЮ на npmjs.com, хранится в GitHub Secrets, живёт пока не отзовёшь, даёт доступ к npm registry.

### 20. Маскировка секретов?
GitHub сканирует весь stdout/stderr и заменяет совпадения с секретами на `***`. Но если секрет закодировать (base64, reverse), маскировка не сработает — поэтому НИКОГДА нельзя echo-ить секреты.

### 21. Зачем `permissions`?
Ограничивает права `GITHUB_TOKEN` по принципу Least Privilege. Без этого блока токен получает ВСЕ права по умолчанию (write на contents, issues, PR и т.д.), что создаёт риск при компрометации.

### 22. Почему `write` в release, `read` в pr-checks?
`pr-checks` только ЧИТАЕТ код (checkout, lint, build, test) — ему `read` достаточно. `release` ПИШЕТ в репо (git tag, git push, gh release create) — ему нужен `write`.

### 23. Least Privilege?
Принцип: давай минимально необходимые права. Если workflow нужен только read — не давай write. Если компрометированная зависимость попытается `git push --force`, с `read` она получит 403 Forbidden.

### 24. CI vs CD?
**CI (Continuous Integration)** — автоматическая проверка кода при каждом push (lint, build, test). **Continuous Delivery** — CI + автоматическая подготовка к релизу, но публикация вручную. **Continuous Deployment** — CI + автоматическая публикация после merge, ноль ручных шагов.

### 25. Какие workflow что реализуют?
CI → `pr-checks.yml` (lint, build, test на каждый PR). Delivery → `label-publish.yml` (артефакт .tgz до мерджа). Deployment → `release.yml` (npm publish автоматически после merge).

### 26. Shift Left?
«Сдвигай проверки влево» = как можно раньше в пайплайне. Пример: проверка версии в `label-publish.yml` (ДО мерджа), а не в `release.yml` (ПОСЛЕ мерджа). Если версия занята, ловим это ещё на стадии PR, а не когда код уже в main.

### 27. Fail Fast?
«Падай на самом раннем, дешёвом этапе». Порядок: Lint (секунды, ловит стиль) → Build (секунды, ловит типы) → Test (медленнее, ловит логику). Если lint упал — нет смысла запускать тесты.

### 28. Flaky test?
Тест, который проходит 99% запусков, но случайно падает 1%. Причины: зависимость от времени, сети, порядка выполнения. Опасен: разработчики перестают доверять CI и начинают игнорировать красные билды.

### 29. Branch Protection?
Набор правил на ветке (обычно `main`), которые БЛОКИРУЮТ merge, если условия не выполнены: обязательные чеки, ревью, актуальность ветки. Настраивается через UI или API.

### 30. `strict: true`?
Требует, чтобы ветка PR была актуальна с main перед merge. Если кто-то смержил изменения в main после того как ты создал PR, тебе нужно обновить свою ветку и дождаться перезапуска тестов.

### 31. Два одновременных PR?
Первый мержится. Второй помечается «out-of-date» благодаря `strict: true`. Автор второго PR должен нажать «Update branch» (GitHub смержит main в его ветку), тесты перезапустятся, и только потом можно мержить.

### 32. `dismiss_stale_reviews`?
Когда разработчик пушит новый коммит после получения аппрува, старый аппрув СБРАСЫВАЕТСЯ. Ревьюер должен посмотреть новые изменения заново. Предотвращает атаку «тихий коммит» (пуш вредоносного кода после одобрения).

### 33. `required_linear_history`?
Запрещает merge commits (с двумя родителями). Разрешает только Squash and Merge (все коммиты PR → один) и Rebase and Merge (коммиты пересаживаются на верхушку main).

### 34. Merge vs Squash vs Rebase?
**Merge commit** — создаёт коммит M с двумя родителями, «вилка» в истории. **Squash** — сжимает все коммиты PR в один, чистая линейная история. **Rebase** — «пересаживает» каждый коммит на верхушку main, линейная история с сохранением всех коммитов (но с новыми SHA).

### 35. Линейная история и git bisect?
`git bisect` — бинарный поиск бага: «на коммите A работало, на Z — нет, найди сломавший коммит». С линейной историей bisect работает за log₂(N) шагов. С merge commits путается на развилках (у merge commit два родителя — какого проверять?).

### 36. Автоматизация Branch Protection?
1) Воспроизводимость — одна команда вместо 15 минут кликов. 2) Масштабирование — 50 репо за цикл `for`. 3) Version control — команду можно положить в скрипт и закоммитить. 4) Задание требует автоматизацию.

### 37. Unit vs E2E?
**Unit** — тестируют отдельные функции из исходников (`src/`), быстрые, ловят логические ошибки. **E2E** — собирают проект (`npm run build`), импортируют из скомпилированного `dist/`, проверяют что готовый пакет работает — имитируют реального пользователя.

### 38. Unit на каждый push, E2E по лейблу — почему?
Unit быстрые (~3 мс), их дешёво запускать часто. E2E медленнее и тяжелее (нужна сборка). В больших проектах E2E могут занимать минуты/часы. Лейбл `verify` — осознанное решение: «код готов к полной проверке».

### 39. E2E из `dist/`, а не `src/`?
Потому что пользователь получает `dist/`, а не `src/`. Между ними стоит компилятор `tsc` с настройками. Ошибка в `tsconfig.json` может привести к тому, что unit-тесты зелёные (src/ в порядке), но скомпилированный пакет не работает.

### 40. Проверка `.d.ts`?
Проверяет, что файл с описанием типов (`index.d.ts`) был сгенерирован при компиляции. Без него пользователь пакета не получит автокомплит и подсказки типов в IDE. Генерируется благодаря `"declaration": true` в `tsconfig.json`.

### 41. `beforeAll()`?
Хук, который выполняется ОДИН РАЗ перед ВСЕМИ тестами в `describe`-блоке. У нас: собирает проект (`npm run build`) перед запуском E2E тестов. Аналог: `beforeEach()` — перед КАЖДЫМ тестом.

### 42. Почему Vitest, а не Jest?
1) Нативная поддержка ES Modules (`"type": "module"` в нашем package.json). Jest требует `--experimental-vm-modules`. 2) Нативная поддержка TypeScript без `ts-jest`. 3) Быстрее (использует Vite для трансформации). 4) API совместим с Jest (`describe`, `it`, `expect`).

### 43. `vitest run`?
Запускает все тесты ОДИН РАЗ (без watch mode). Vitest читает `vitest.config.ts`, находит файлы по паттерну `include` (по умолчанию `**/*.test.ts`), трансформирует TS→JS через Vite, выполняет каждый `it()` блок, собирает результаты.

### 44. Scoped package?
Пакет с префиксом `@scope/name`. Привязан к аккаунту/организации. `@emilmamedov-inno/ts-math-utils` может опубликовать только аккаунт `emilmamedov-inno`. Решает проблему захвата имён (name squatting).

### 45. `--access public`?
По умолчанию scoped-пакеты приватные (npm Pro, $7/мес). `--access public` делает пакет доступным для всех бесплатно.

### 46. `package-lock.json`?
Фиксирует ТОЧНЫЕ версии ВСЕХ зависимостей (включая зависимости зависимостей). `package.json` допускает диапазоны (`^3.0.0` = любая 3.x). Без lock-файла разные машины/дни могут получить разные версии → нестабильный билд.

### 47. `npm ci` vs `npm install`?
`npm ci`: удаляет `node_modules`, читает `package-lock.json`, ставит ТОЧНО указанные версии, падает при несовпадении lock с package.json. `npm install`: читает `package.json`, МОЖЕТ обновить lock. В CI/CD `npm ci` — единственный правильный выбор (детерминированность).

### 48. SemVer?
`MAJOR.MINOR.PATCH`. PATCH (1.0.0→1.0.1): баг-фикс. MINOR (1.0.0→1.1.0): новая фича, обратно совместимо. MAJOR (1.0.0→2.0.0): breaking change, ломает чужой код.

### 49. Explicit bump?
Только разработчик знает СЕМАНТИКУ изменения (фикс, фича, ломающее). Автоматические инструменты (Semantic Release) могут угадать из commit messages, но требуют Conventional Commits. Задание требует explicit — ручное изменение `"version"` в `package.json`.

### 50. Суффикс `-dev-<sha>`?
`1.1.0-dev-a3b8c2f` — pre-release версия для release candidate. `dev` = не финальная. `a3b8c2f` = первые 7 символов SHA коммита (уникальный ID). По SemVer prerelease меньше финальной: `1.1.0-dev-xxx < 1.1.0`. Не установится автоматически при `npm install`.

### 51. `npm pack`?
Создаёт `.tgz` архив — ТОЧНО то, что получит пользователь при `npm install`. Артефакт до мерджа позволяет скачать и проверить содержимое вручную: все ли файлы на месте, правильная ли версия.

### 52. Забыли поднять версию?
Workflow `label-publish.yml` проверяет через `check-version-exists` action: версия уже в npm? Если да → `exit 1` → workflow красный → merge заблокирован Branch Protection. Разработчик видит ошибку и должен поднять версию.

### 53. Условие запуска публикации?
```yaml
if: github.event.pull_request.merged == true &&
    contains(github.event.pull_request.labels.*.name, 'publish')
```
PR должен быть СМЕРЖЕН (не просто закрыт) И иметь лейбл `publish`. Оба условия одновременно.

### 54. `fetch-depth: 0`?
По умолчанию `actions/checkout` скачивает только ПОСЛЕДНИЙ коммит (shallow clone). `fetch-depth: 0` скачивает ВСЮ историю. Нужно для: 1) `git tag` — правильное создание тега. 2) `gh release --generate-notes` — составить changelog между тегами.

### 55. Git Tag?
Неподвижный указатель на конкретный коммит. Annotated (`git tag -a`) — содержит автора, дату, сообщение, можно подписать GPG. Lightweight (`git tag`) — просто ссылка на SHA. Для релизов используют annotated.

### 56. `gh release create --generate-notes`?
1) Находит предыдущий тег. 2) Вычисляет `git log prev..current`. 3) Для каждого коммита ищет связанный PR. 4) Генерирует markdown со списком PR и авторов. 5) Создаёт страницу Release на GitHub.

### 57. Полный путь кода?
1) `git push` в feature-ветку. 2) Создание PR → `pr-checks.yml`: lint, build, unit tests. 3) Лейбл `verify` → `label-verify.yml`: build + E2E тесты. 4) Лейбл `publish` → `label-publish.yml`: проверка версии, RC с dev-суффиксом, .tgz артефакт. 5) Ревью + Approve. 6) Squash and Merge. 7) `release.yml`: build → npm publish → git tag → GitHub Release.

### 58. Зачем TypeScript?
Статическая типизация: компилятор проверяет типы ДО запуска. Ловит ошибки, которые JavaScript пропустил бы (неправильный тип аргумента, отсутствующее свойство). Генерирует `.d.ts` для IDE-подсказок пользователям пакета.

### 59. `.d.ts` файлы?
Declaration files — описание типов без реализации. Когда пользователь установит пакет, его IDE прочитает `.d.ts` и покажет: «функция sum принимает два числа и возвращает число». Генерируются при `"declaration": true` в tsconfig.

### 60. Этапы `tsc`?
1) Парсинг — построение AST (дерево кода). 2) Type Checking — проверка совместимости типов. 3) Emit — генерация `.js` и `.d.ts`. 4) Module Resolution — вычисление путей к `import`.

### 61. Type Erasure?
Типы СТИРАЮТСЯ при компиляции в JavaScript. `function sum(a: number, b: number): number` → `function sum(a, b)`. В рантайме (при выполнении) типов нет. TypeScript — чисто compile-time инструмент.

### 62. Что ловит компилятор, а не линтер/тесты?
1) Присвоение лишнего свойства объекту (structural typing). 2) Передачу строки вместо массива в generic-функцию. 3) Неполный switch по union-типу (exhaustive check). Линтер не видит — он не проверяет типы. Тесты не всегда покрывают edge-case, а в JS код часто «молча работает» неправильно.

### 63. Линтер vs тесты?
Линтер — **статический** анализ: читает текст кода, ищет подозрительные паттерны, НЕ запускает код. Тесты — **динамический** анализ: запускают код и проверяют результат. Линтер мгновенный, тесты могут быть медленными.

### 64. AST?
Abstract Syntax Tree — дерево, описывающее структуру кода. `const x = 5` → узел VariableDeclaration с именем «x» и значением NumericLiteral(5). Каждое правило ESLint — функция-визитор, обходящая это дерево и ищущая нарушения.

### 65. Flat Config?
Новый формат конфигурации ESLint (v9+). Вместо `.eslintrc.json` (каскад конфигов в каждой папке, сложно понять какой победил) — один файл `eslint.config.js`, экспортирующий массив объектов. Предсказуемо, явно, обычный JS-модуль.

### 66. NPM_TOKEN протух?
`npm publish` вернёт 401 Unauthorized. Workflow упадёт. Код УЖЕ в main (merge произошёл). Нужно: обновить токен в GitHub Secrets → перезапустить workflow (Actions → Re-run), или создать trivial PR для повторного мерджа.

### 67. Два PR с `publish` одновременно?
Оба запустят `release.yml`. Если версия одинаковая — первый опубликует, второй получит 403 от npm. Если версии разные — оба опубликуются. Решение: `concurrency` group в workflow (мы не используем, но в продакшене стоит).

### 68. PR без лейбла `publish`?
PR можно смержить (если чеки зелёные и есть аппрув). Код попадёт в main. Но `release.yml` НЕ запустится — условие `contains(..., 'publish')` = false. Пакет НЕ опубликуется, тег НЕ создастся. Это нормально — не каждый PR = релиз.

### 69. Зелёные тесты, но сломалось после merge?
Возможно если: 1) Два PR прошли проверку одновременно и их изменения конфликтуют (strict: true снижает риск, но не убирает на 100%). 2) Внешняя зависимость изменилась. 3) Flaky test. Решение: Merge Queue.

### 70. Зачем build дублируется?
Runner между workflows НЕ shared — это РАЗНЫЕ виртуальные машины. Артефакты build из `pr-checks` не переносятся в `release`. В `pr-checks` build нужен для ПРОВЕРКИ («компилируется ли?»). В `release` — для СБОРКИ перед публикацией.
